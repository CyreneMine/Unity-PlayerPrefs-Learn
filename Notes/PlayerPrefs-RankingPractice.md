# PlayerPrefs 多玩家和排行榜练习笔记

## 本节目标

本节练习在上一节玩家数据保存的基础上继续扩展：

- 让同一套玩家字段可以保存多名玩家
- 用 PlayerPrefs 保存排行榜数据
- 排行榜记录包含玩家名、玩家得分、玩家通关时间

## 多玩家数据保存

PlayerPrefs 是 key-value 存储。如果多个玩家都使用相同 key，例如：

```text
name
age
atk
def
```

那么后保存的玩家会覆盖前一个玩家的数据。

本次练习的处理方式是给 key 加上玩家前缀：

```text
Player1_name
Player1_age
Player1_atk
Player1_def

Player2_name
Player2_age
Player2_atk
Player2_def
```

这样同一套保存逻辑可以复用，只要传入不同的 `keyName`，就能区分不同玩家的数据。

## 排行榜保存思路

排行榜本质上也是列表数据。当前练习使用：

```text
rankListNum
rankInfo0
rankScore0
rankTime0
rankInfo1
rankScore1
rankTime1
```

读取时先读取 `rankListNum`，再根据下标循环恢复每一条 `RankInfo`。

## 关于重复玩家名

观看教程答案时发现，教程没有处理重复玩家名。如果直接追加，会出现同一个玩家多次出现在排行榜里的情况。

这不一定是语法错误，但从“排行榜”的产品逻辑看，一般不太合理。排行榜通常更像是“每个玩家当前的一条成绩记录”，而不是“玩家每一次游玩的历史记录”。

本次练习选择的规则是：

- 如果排行榜中不存在该玩家名，则新增记录
- 如果排行榜中已经存在该玩家名，则不再新增重复记录
- 如果新分数更高，则更新这个玩家的排行榜成绩

这个修改比单纯照抄教程更接近真实需求，因为它先明确了排行榜的数据规则。

## 本次修正记录

- 多玩家读取装备列表时，在 `Load()` 开头使用 `items.Clear()` 清空旧列表，避免同一个 `Player` 对象重复调用 `Load()` 时把装备重复追加进去。
- 排行榜展示中补充输出通关时间，方便验证题目要求中的三项数据是否都被保存和读取。
- 排行榜同名玩家不再重复追加，而是按当前规则更新已有记录。

## 后续可以继续优化的地方

- 当前规则已经偏向“排行榜保留更高分”。如果新分数更高但通关时间更慢，需要明确是否把分数和时间作为同一次通关记录一起更新。
- 如果分数相同，可以用通关时间作为第二排序条件，时间越短排名越靠前。
- 当前排序主要按分数排序，后续可以在分数相同的时候比较通关时间。
- 测试 Lesson2 时，如果 Lesson1 也挂在同一个场景里，要注意两个脚本可能同时输出 Console 日志；可以临时禁用不测试的脚本组件。
