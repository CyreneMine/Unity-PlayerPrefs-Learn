# 结合反射保存基础类型 List

## 本节目标

在已有基础类型保存逻辑上增加 List 保存支持。目前列表元素只处理 `int`、`float`、`string` 和 `bool`，自定义类型将在后续补全。

## 判断 List 类型

当前使用：

```csharp
typeof(IList).IsAssignableFrom(fieldType)
```

它判断当前字段类型能否赋值给 `IList`。`List<T>` 实现了 `IList`，因此可以通过该判断进入列表保存分支。

使用非泛型 `IList` 后，管理器不需要提前知道具体的 `T`，也能读取列表数量并遍历其中的元素。

## 保存结构

List 无法直接交给 PlayerPrefs 保存，因此当前拆成两部分：

```text
列表Key        -> 列表元素数量
列表Key0       -> 第 0 个元素
列表Key1       -> 第 1 个元素
列表Key2       -> 第 2 个元素
```

保存数量后，读取阶段才能知道应该恢复多少个元素。

## 复用 `SaveValue`

遍历列表时，每个元素继续调用：

```csharp
SaveValue(obj, keyName + index);
```

这样 List 分支只负责数量和下标，元素的类型判断仍由 `SaveValue` 统一负责。

当前元素为基础类型时，会进入已有的 `int`、`float`、`string` 或 `bool` 分支。后续为 `SaveValue` 增加自定义类型处理后，List 分支也可以继续复用它。

## 当前测试

测试类增加了：

```csharp
public List<int> list = new List<int>() { 1, 2, 3, 4 };
```

它验证了列表数量和 `int` 元素保存流程。其他基础类型 List 会复用相同分支，但仍可以分别添加测试样例，确认保存结果符合预期。

## 边界与后续处理

### 空列表

空列表不是 `null`，会进入 List 分支并保存数量 `0`，这是合理结果。

### 列表缩短

如果上次保存 4 个元素，本次只保存 2 个，新数量会变成 2，但旧的第 2、3 项 key 仍可能留在存储中。读取时按新数量处理可以忽略旧项，是否删除残留 key 将在框架完成后统一设计。

### `null` 元素

列表数量包含 `null` 元素的位置，但 `SaveValue` 会跳过该元素，因此会产生有下标却没有对应保存值的情况。后续需要明确：

- List 是否允许 `null` 元素
- 使用特殊标记保存 `null`
- 或保存时过滤 `null` 并调整数量

这个规则应与普通字段的 `null` 和旧 key 清理策略统一决定。

## 本节理解

保存 List 的重点不是把 List 本身塞进 PlayerPrefs，而是保存“列表结构信息”和“每个元素”。通过递归复用 `SaveValue`，列表保存逻辑可以与元素类型处理解耦。
